# Story 1.4.2 — Remover todos os `any` dos mappers TypeScript

**Status:** Done | **Sprint 4 — Quality & DevOps** ✅ | **Agente:** @dev | **Esforço:** 2h

---

## Story Statement

**Como** desenvolvedor do Nexus OS,
**Quero** eliminar todos os tipos `any` explícitos do codebase TypeScript,
**Para que** o TypeScript forneça proteção real de tipos e erros sejam capturados em tempo de compilação.

---

## Contexto

O TypeScript com `any` é essencialmente JavaScript com sintaxe extra — a segurança de tipos é perdida. Os `any` foram identificados principalmente nos mappers de resposta do Supabase e em interfaces de componentes.

[Source: docs/architecture/technical-debt-assessment.md — Qualidade de Código 4/10]

---

## Acceptance Criteria

## Dev Notes

### Padrões comuns de substituição

```typescript
// Supabase query response
// ❌ const { data }: { data: any } = await supabase...
// ✅ const { data }: { data: Lead[] | null } = await supabase...

// Event handler
// ❌ onChange={(e: any) => setValue(e.target.value)}
// ✅ onChange={(e: React.ChangeEvent<HTMLInputElement>) => setValue(e.target.value)}

// Props genéricos
// ❌ function Component({ data }: { data: any })
// ✅ function Component<T>({ data }: { data: T })
```

### Casos onde `any` pode ser mantido com justificativa

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const config = JSON.parse(rawConfig) as any  // JSON externo sem schema conhecido
```
