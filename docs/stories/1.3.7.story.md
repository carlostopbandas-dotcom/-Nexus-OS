## Story Statement

**Como** Carlos (CEO / único usuário do Nexus OS),
**Quero** que as queries mais frequentes do dashboard tenham índices de banco de dados,
**Para que** o app continue rápido à medida que os dados crescem com meses de uso.

---

## Contexto

Hoje nenhuma tabela tem índices além da PK (uuid). Com dados pequenos (<1000 rows), não é um problema visível. Com 6-12 meses de dados de negócio (leads, calls, tasks), cada query frequente será um full table scan.

Débito remediado: **NEW-DB-3** (ausência de índices compostos para access patterns reais).

## Dev Notes

### SQL completo

```sql
-- supabase/migrations/004_performance_indexes.sql

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_leads_created_at
  ON leads (created_at DESC);

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_leads_status
  ON leads (status) WHERE status IS NOT NULL;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_tasks_pending
  ON tasks (created_at DESC) WHERE completed = false;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_call_logs_created_at
  ON call_logs (created_at DESC);

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_call_logs_lead_id
  ON call_logs (lead_id) WHERE lead_id IS NOT NULL;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_store_metrics_date
  ON store_metrics (date DESC);

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_events_day_offset
  ON events (day_offset) WHERE day_offset IS NOT NULL;

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_okrs_key_results_gin
  ON okrs USING GIN (key_results);
```

[Source: docs/architecture/db-specialist-review.md#34-migration-004--performance-indexes]

### Partial indexes

Os partial indexes (com `WHERE`) são mais eficientes porque indexam apenas o subconjunto de rows relevante:
- `idx_tasks_pending`: apenas tasks `completed = false` — as que aparecem na UI principal
- `idx_leads_status`: apenas leads com status preenchido
- `idx_call_logs_lead_id`: apenas calls linkadas a leads
