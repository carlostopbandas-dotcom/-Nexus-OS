# Story 2.6 — Visibilidade de campanhas ativas com status lucratividade

## Story

**Como** CEO ou gestor de loja,  
**Quero** ver o status de lucratividade de cada campanha de Ads ativa no mesmo dashboard financeiro,  
**Para** identificar rapidamente se as campanhas estão acima ou abaixo da meta de ROAS sem cruzar dados manualmente.

## Context

Stories 2.4 e 2.5 entregaram o Dashboard Financeiro com ROAS por loja (nível agregado) e registro de lançamentos financeiros. O Sprint 3 fecha com esta story: tornar visível, dentro do mesmo contexto de loja/período selecionado, quais campanhas de Ads estão rodando e se a loja está dentro da meta de ROAS.

O ROAS não é calculado por campanha individual (a tabela `ads_campaigns_cache` não contém receita por campanha — somente gasto). O ROAS da loja (`store_metrics.sales / ads_campaigns_cache.spend`) é calculado pelo `financialService` existente e atribuído como status agregado de todas as campanhas daquela loja.

Nenhuma migration necessária — utiliza tabelas e RLS entregues em 2.2 (`ads_campaigns_cache`) e 2.4 (`store_metrics`).

## Dev Notes

### Arquitetura de dependências
```
CampaignsProfitabilityPanel
  └── campaignProfitabilityService
        ├── financialService.getStoreFinancials()  ← ROAS da loja (2 queries: store_metrics + ads_campaigns_cache)
        └── adsService.getCampaigns()              ← lista de campanhas com gasto e CTR (1 query: ads_campaigns_cache)
```

**Double-query em `ads_campaigns_cache`:** `financialService.getStoreFinancials` busca a coluna `spend` para somar o adSpend total; `adsService.getCampaigns` busca `*` para a lista detalhada. São queries distintas ao mesmo nível de filtro (`store_name`, `date_start`, `date_end`). Aceitável no escopo atual — otimizável em sprint futura com query consolidada.

### Ordem dos mocks nos testes

`getStoreSummary` usa internamente:
```typescript
const [financialsResult, campaignsResult] = await Promise.all([
  financialService.getStoreFinancials(store, period),   // faz 2 queries internas
  adsService.getCampaigns(store, dateRange),            // faz 1 query
])
```

Sequência de `supabase.from()` calls por store:
1. `store_metrics` (dentro de `getStoreFinancials` → `Promise.all` interno, 1ª query)
2. `ads_campaigns_cache` — spend agregado (dentro de `getStoreFinancials`, 2ª query)
3. `ads_campaigns_cache` — lista detalhada (dentro de `getCampaigns`)

Total: **3 `mockReturnValueOnce` por loja**. Para `getGroupSummary` com 3 lojas: **9 calls**.

Exemplo para teste `getStoreSummary profitable`:
```typescript
// 1) store_metrics
mockFrom.mockReturnValueOnce(createMockBuilder({ data: [{ sales: 9000 }], error: null }))
// 2) ads_campaigns_cache (financialService — spend para ROAS)
mockFrom.mockReturnValueOnce(createMockBuilder({ data: [{ spend: 3000 }], error: null }))
// 3) ads_campaigns_cache (adsService — lista campanhas)
mockFrom.mockReturnValueOnce(createMockBuilder({ data: [{ id: 'c1', campaign_name: 'Campanha A', spend: 3000, impressions: 10000, clicks: 300, ... }], error: null }))
```
