# Story 1.1.4 — Documentar `.env.example` com todas as variáveis

**Status:** Ready | **Sprint 1 — Security Foundation** 🔴 | **Agente:** @dev | **Esforço:** 30min

---

## Story Statement

**Como** desenvolvedor (atual ou futuro) do Nexus OS,
**Quero** ter um arquivo `.env.example` com todas as variáveis de ambiente necessárias para rodar o projeto,
**Para que** o setup de um novo ambiente (dev, staging, CI) seja possível sem precisar perguntar quais variáveis existem ou procurar no código.

---

## Contexto

Após as Stories 1.0.2 e 1.1.2, o projeto usa variáveis de ambiente para credenciais sensíveis. Mas sem um `.env.example`, um novo desenvolvedor (ou uma pipeline de CI) não sabe quais variáveis precisam ser configuradas.

O `.env.example` é uma convenção padrão: contém todas as chaves com valores de placeholder (sem valores reais), e é commitado no repositório. O `.env.local` (com valores reais) permanece gitignored.

## Dev Notes

### Estrutura esperada do `.env.example`

```bash
# ============================================================
# Nexus OS — Variáveis de Ambiente
# Copie este arquivo para .env.local e preencha os valores reais
# NUNCA commite .env.local — ele está no .gitignore
# ============================================================

# ---- Supabase -----------------------------------------------
# URL do projeto Supabase: Supabase Dashboard → Project Settings → API
VITE_SUPABASE_URL=https://seu-projeto.supabase.co

# Anon key pública (safe para client-side com RLS ativo)
# Supabase Dashboard → Project Settings → API → Project API keys
VITE_SUPABASE_ANON_KEY=sb_publishable_xxxxxxxxxxxx

# ---- Google Gemini AI ----------------------------------------
# API key para Gemini — usada APENAS pelo servidor (Edge Function)
# Não deve aparecer no bundle frontend após Story 1.1.2
# Obter em: https://aistudio.google.com/apikey
VITE_GEMINI_API_KEY=AIzaxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
# Nota: após Story 1.1.2, esta var só é necessária como Supabase Secret
# no dashboard Edge Functions, não mais como VITE_ var no frontend

# ---- Supabase Edge Functions (server-side) -------------------
# Estas variáveis são configuradas como Supabase Secrets, não como .env.local
# Documentadas aqui apenas para referência:
# GEMINI_API_KEY → configurar via: supabase secrets set GEMINI_API_KEY=<valor>
```

### Nota sobre VITE_ prefix

O Vite expõe automaticamente variáveis com prefixo `VITE_` via `import.meta.env`. Variáveis sem este prefixo não são expostas ao frontend — são server-only.

Para as Supabase Edge Functions, as variáveis são configuradas via `supabase secrets set`, não via `.env.local`.
