# Story 1.2.4 — Substituir event bus por store subscriptions

**Status:** Done | **Sprint 2 — Architecture & DB Foundation** 🟢 | **Agente:** @dev | **Esforço:** 3h

---

## Story Statement

**Como** desenvolvedor do Nexus OS,
**Quero** remover o event bus `window.dispatchEvent('nexus-data-updated')` e substituir por atualizações diretas no Zustand store,
**Para que** o fluxo de dados seja previsível, testável e não dependa de eventos globais no DOM.

---

## Contexto

O padrão atual de sincronização de dados é:
1. Página faz mutação (ex: criar lead)
2. Dispara `window.dispatchEvent(new CustomEvent('nexus-data-updated'))`
3. App.tsx escuta o evento e chama `fetchData()` (re-busca tudo)

Problemas:
- Anti-pattern: event bus via DOM é imprevisível
- Ineficiente: qualquer mudança re-fetcha todos os dados
- Não testável: difícil simular eventos DOM em testes
- Acoplamento invisível: difícil rastrear quem dispara/escuta

Débito remediado: **TD-H4** (event bus via window.dispatchEvent).

## Dev Notes

### Fluxo atual vs target

```
ATUAL (event bus):
Componente → supabase.insert(lead) → window.dispatchEvent('nexus-data-updated')
                                              ↓
                                      App.tsx listener
                                              ↓
                                      fetchData() — 7 queries

TARGET (store):
Componente → leadsService.create(lead) → useAppStore.getState().addLead(newLead)
                                                ↓
                                      UI reativa via Zustand (0 queries extras)
```

[Source: docs/architecture/system-architecture.md#data-flow]

### Padrão de mutation action no store

```typescript
// Adições ao useAppStore.ts
addLead: (lead: Lead) => set(state => ({
  leads: [lead, ...state.leads]
})),

updateLead: (id: string, updates: Partial<Lead>) => set(state => ({
  leads: state.leads.map(l => l.id === id ? { ...l, ...updates } : l)
})),

removeLead: (id: string) => set(state => ({
  leads: state.leads.filter(l => l.id !== id)
})),
```

### Caso especial: VoiceAssistant log_sales_call

O VoiceAssistant usa function calling do Gemini para inserir call_logs. Após esta story, a function `log_sales_call` deve:
1. Inserir no Supabase via `callLogsService.create()`
2. Chamar `useAppStore.getState().addCallLog(newCallLog)` diretamente

```typescript
// VoiceAssistant.tsx — padrão target
async function handleLogSalesCall(callData: CallLogInput) {
  const { data, error } = await callLogsService.create(callData)
  if (data) useAppStore.getState().addCallLog(data)
}
```

### Optimistic updates existentes

O Pipeline.tsx já usa optimistic updates (move lead entre colunas instantaneamente, rollback em erro). Este padrão deve ser preservado com o store:

```typescript
// Pipeline — padrão optimistic com store
const { updateLead } = useAppStore()

async function moveLead(id: string, newStatus: LeadStatus) {
  updateLead(id, { status: newStatus })  // optimistic
  const { error } = await leadsService.update(id, { status: newStatus })
  if (error) updateLead(id, { status: previousStatus })  // rollback
}
```
