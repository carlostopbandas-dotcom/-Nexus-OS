# Story 1.2.6 — Script de backfill `call_logs.lead_name → lead_id`

**Status:** Ready | **Sprint 2 — Architecture & DB Foundation** 🟠 | **Agente:** @data-engineer | **Esforço:** 1h

---

## Story Statement

**Como** Carlos (CEO / único usuário do Nexus OS),
**Quero** que os registros existentes de `call_logs` tenham o `lead_id` preenchido a partir do `lead_name`,
**Para que** a relação entre calls e leads seja baseada em FK real, não em texto livre sujeito a erros de digitação.

---

## Contexto

Após Story 1.2.5, a coluna `call_logs.lead_id` foi criada mas está vazia para todos os registros existentes. O campo `call_logs.lead_name` contém o nome do lead como texto livre — precisa ser matchado para o `leads.id` correspondente.

Este é um script de backfill único (não uma migration recorrente). Registros sem match permanecem com `lead_id = NULL`.

[Source: docs/architecture/db-specialist-review.md#agr-1]
[Source: docs/architecture/SCHEMA.md#call_logs]

## Acceptance Criteria

- [ ] **AC-1:** Script SQL de backfill criado em `supabase/scripts/backfill_call_logs_lead_id.sql`
- [ ] **AC-2:** Script faz match por nome (case-insensitive) entre `call_logs.lead_name` e `leads.name`
- [ ] **AC-3:** `call_logs.lead_id` preenchido para todos os registros com match encontrado
- [ ] **AC-4:** Registros sem match mantêm `lead_id = NULL` (sem erro)
- [ ] **AC-5:** Script gera relatório: quantos registros foram matchados e quantos ficaram sem match
- [ ] **AC-6:** `call_logs.lead_name` mantido (não deletado) — coluna nova por compatibilidade com código existente

---

## Tasks / Subtasks

- [ ] **Task 1 — Criar script de backfill** (AC: 1, 2, 3, 4, 5)
  ```sql
  -- supabase/scripts/backfill_call_logs_lead_id.sql
  BEGIN;
  UPDATE call_logs cl
  SET lead_id = l.id
  FROM leads l
  WHERE LOWER(cl.lead_name) = LOWER(l.name)
    AND cl.lead_id IS NULL;

  -- Relatório
  SELECT
    COUNT(*) FILTER (WHERE lead_id IS NOT NULL) AS matched,
    COUNT(*) FILTER (WHERE lead_id IS NULL) AS unmatched,
    COUNT(*) AS total
  FROM call_logs;

  COMMIT;
  ```

- [ ] **Task 2 — Executar em produção** (AC: 3, 4)
  - Via Supabase Dashboard → SQL Editor
  - Anotar resultado do relatório (matched vs unmatched)

- [ ] **Task 3 — Tratar unmatcheds manualmente** (AC: 4)
  - Listar calls sem match: `SELECT id, lead_name FROM call_logs WHERE lead_id IS NULL`
  - Para cada uma: verificar se o lead existe com nome diferente e corrigir manualmente se necessário

- [ ] **Task 4 — Commit** (AC: 1)
  - `git add supabase/scripts/backfill_call_logs_lead_id.sql`
  - `git commit -m "feat: script de backfill call_logs.lead_id [Story 1.2.6]"`

## Dev Notes

### Riscos do match por nome

O `lead_name` em `call_logs` foi digitado manualmente ou inserido pelo VoiceAssistant. Pode haver:
- "João Silva" vs "joao silva" → resolvido por LOWER()
- "João" vs "João Silva" → sem match (lead_id NULL — tratamento manual)
- Typos: "Julia" vs "Júlia" → sem match automático

Para typos, usar fuzzy match manual após ver o relatório.

[Source: docs/architecture/db-specialist-review.md#agr-1]

### Nota: lead_name não será removido agora

O código do app ainda usa `lead_name` em vários lugares. Remover a coluna requer atualizar o código primeiro. Esta story apenas popula `lead_id` — a remoção de `lead_name` é uma story futura (pós Sprint 2).
