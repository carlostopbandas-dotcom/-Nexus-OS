# Story 1.0.1 — Capturar schema atual em `001_initial_schema.sql`

**Status:** Ready | **Sprint 0** | **Agente:** @data-engineer | **Esforço:** 30min

> **Bloqueante crítico:** Deve ser a PRIMEIRA ação do EPIC-1. Sem migrations versionadas, qualquer rollback de schema é impossível. Nenhum outro trabalho de desenvolvimento deve prosseguir até este passo estar concluído.

---

## Story Statement

**Como** Carlos (CEO / único usuário do Nexus OS),
**Quero** que o schema atual do banco de dados seja capturado em um arquivo de migration versionado (`001_initial_schema.sql`),
**Para que** o estado atual do banco esteja documentado, seja possível recriar o banco do zero, e todas as mudanças futuras tenham uma baseline a partir da qual se derivam.

---

## Contexto

Hoje o Nexus OS não possui nenhum sistema de versionamento de schema. O banco Supabase foi criado manualmente via dashboard, sem migrations. Isso significa:

- Rollback de schema é impossível
- Onboarding de novo ambiente (staging, dev) requer recriação manual
- Não há como auditar quando cada tabela foi criada ou modificada

Este é o débito **TD-C5** identificado no Brownfield Discovery (classificado CRÍTICO).

[Source: docs/architecture/technical-debt-assessment.md#td-c5]
[Source: docs/architecture/db-specialist-review.md#migration-plan]

## Dev Notes

### Schema completo a capturar

Todas as tabelas são extraídas do `SCHEMA.md` (Brownfield Phase 2). O SQL exato está em `db-specialist-review.md` Seção 3.1.

**Tabela `leads`:**
```sql
CREATE TABLE IF NOT EXISTS leads (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name        text NOT NULL,
  email       text,
  source      text,
  status      text,
  value       numeric,
  product     text,
  created_at  timestamptz NOT NULL DEFAULT now()
);
```

**Tabela `call_logs`** (atenção: `lead_name` é TEXT livre — não FK):
```sql
CREATE TABLE IF NOT EXISTS call_logs (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  lead_name           text NOT NULL,
  date                text,
  duration            text,
  type                text,
  status              text,
  sentiment           text,
  transcript_snippet  text,
  summary             text,
  recording_url       text,
  created_at          timestamptz DEFAULT now()
);
```

> O SQL completo das 7 tabelas está em [Source: docs/architecture/db-specialist-review.md#31-migration-001--initial-schema-capture]

### Importante: NÃO corrigir issues nesta story

Esta story captura o estado atual **exatamente como está**. Issues como:
- `email` nullable em `leads` → corrigir em Story 1.2.5
- `lead_name` TEXT (sem FK) → corrigir em Story 1.2.6
- Ausência de `updated_at` → corrigir em Story 1.2.5
- Sem índices → corrigir em Story 1.3.7

**Qualquer correção de schema nesta story é OUT OF SCOPE.**
