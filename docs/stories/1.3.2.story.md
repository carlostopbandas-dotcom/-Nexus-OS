# Story 1.3.2 — Migrar Tailwind CDN → npm + `tailwind.config.js`

**Status:** Ready | **Sprint 3 — Frontend System** 🟡 | **Agente:** @dev | **Esforço:** 2h

> **Dependência de ordem:** APÓS 1.3.1 (baseline), ANTES de 1.3.3 (tokens) e 1.3.4 (shadcn). shadcn exige Tailwind npm.

---

## Story Statement

**Como** desenvolvedor do Nexus OS,
**Quero** migrar o Tailwind CSS de CDN para npm com purging,
**Para que** o bundle seja reduzido em ~400KB, o build seja controlável, e seja possível instalar shadcn/ui.

---

## Contexto

Tailwind via CDN inclui TODAS as classes (sem purge), gerando ~400KB desnecessário no bundle. A CDN também bloqueia a instalação do shadcn/ui (que requer Tailwind npm configurado).

Débito remediado: **TD-C6** (Tailwind CDN), **UX-CRIT-1** (bundle ~400KB extra).
ADR validado: **ADR-5** (Tailwind v3 npm, não v4 — v4 ainda instável para brownfield).

[Source: docs/architecture/technical-debt-assessment.md#td-c6]
[Source: docs/architecture/ux-specialist-review.md#adrs — ADR-5 APROVADO]

### ADR-5: Por que Tailwind v3 e não v4

[Source: docs/architecture/ux-specialist-review.md#adrs]
> "ADR-5: Tailwind v3 npm (não v4) — APROVADO. v4 ainda instável para brownfield."

Tailwind v4 tem mudanças breaking na sintaxe de configuração. Para um projeto brownfield com muitas classes existentes, v3 é a migração segura.

### Possível issue: classes com [] (arbitrary values)

O Nexus OS usa many arbitrary values (`text-[8px]`, `text-[9px]`, `bg-[#F0F4F8]`, `rounded-[1.5rem]`, etc.). Tailwind v3 npm suporta esses valores — verificar que todos continuam funcionando após a migração.

[Source: docs/architecture/ux-specialist-review.md#agr-ux-1]
