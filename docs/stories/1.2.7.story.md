# Story 1.2.7 — Adicionar soft delete em leads e call_logs

**Status:** Done | **Sprint 2 — Architecture & DB Foundation** 🟢 | **Agente:** @data-engineer | **Esforço:** 1h

---

## Story Statement

**Como** Carlos (CEO / único usuário do Nexus OS),
**Quero** que a deleção de leads e calls seja soft delete (marca como deletado, não apaga),
**Para que** seja possível recuperar dados excluídos acidentalmente, mantendo o histórico de negócios.

---

## Contexto

Hoje qualquer deleção de lead ou call_log é permanente e irrecuperável. Leads e call_logs são ativos de negócio críticos — sem soft delete, um erro de clique pode destruir dados de pipeline de vendas sem possibilidade de recovery.

Débito remediado: **TD-C7** (sem soft delete em leads/call_logs) e **NEW-DB-1** (zero observabilidade de dados).

[Source: docs/architecture/technical-debt-assessment.md#td-c7]
[Source: docs/architecture/db-specialist-review.md#51-recomendacao-soft-delete]

  ALTER TABLE call_logs ADD COLUMN IF NOT EXISTS deleted_at timestamptz;

    SELECT * FROM leads WHERE deleted_at IS NULL;

- [x]**Task 3 — Atualizar leadsService.delete()** (AC: 4, 6)
  ```typescript
  // leadsService.ts — antes
  async delete(id: string) {
    return supabase.from('leads').delete().eq('id', id)
  }

  // leadsService.ts — depois (soft delete)
  async delete(id: string) {
    return supabase.from('leads')
      .update({ deleted_at: new Date().toISOString() })
      .eq('id', id)
      .select().single()
  }

  // getAll — filtrar deletados
  async getAll() {
    return supabase.from('leads')
      .select('*')
      .is('deleted_at', null)
      .order('created_at', { ascending: false })
  }
  ```

## Dev Notes

### Por que apenas leads e call_logs

Leads e call_logs são os dados de negócio mais críticos (pipeline de vendas, histórico de calls). As demais tabelas (tasks, events, okrs) têm menor risco de perda irrecuperável e podem ter soft delete adicionado futuramente se necessário.

[Source: docs/architecture/db-specialist-review.md#51-recomendacao-soft-delete]

### RLS e views

Após Story 1.1.3 (RLS ativo), as views herdam as policies das tabelas base. Não é necessário adicionar RLS às views separadamente.

### Numbering da migration

Se a Story 1.2.5 (003_schema_hardening) ainda não foi aplicada em produção, pode-se incluir o soft delete nela. Se já foi aplicada, criar `003b_soft_delete.sql` ou `004_soft_delete.sql` (verificar com DB specialist antes).
