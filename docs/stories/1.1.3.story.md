# Story 1.1.3 — Criar migration `002_security_foundation.sql` (RLS + policies)

**Status:** Ready | **Sprint 1 — Security Foundation** 🔴 | **Agente:** @data-engineer | **Esforço:** 3h

> **Bloqueante crítico:** Esta migration DEVE ser executada apenas APÓS Story 1.1.1 (Supabase Auth implementado). Se RLS for habilitado antes de auth.uid() retornar um UUID válido, TODOS os dados ficam inacessíveis ao app (quebra total).

---

## Story Statement

**Como** Carlos (CEO / único usuário do Nexus OS),
**Quero** que Row Level Security seja habilitado em todas as tabelas do banco com políticas que exigem autenticação para acesso,
**Para que** mesmo que a anon key seja obtida por terceiros, nenhum dado seja acessível sem uma sessão autenticada válida.

---

## Contexto

Hoje o banco Supabase tem RLS desabilitado em todas as tabelas. Isso significa que a anon key exposta + zero RLS = acesso total ao banco de produção para qualquer pessoa.

Após Story 1.1.1, `auth.uid()` retorna um UUID válido para o usuário logado. Com esse UUID disponível, podemos criar políticas RLS que só permitem acesso a sessões autenticadas.

**Vetor de ataque atual (fechado por esta story):**
```bash
curl -H "apikey: sb_publishable_BD0fsd916WaTIoCnLjT6pQ_tQozy6tz" \
     https://zfmfevpdcwcwatpkttnr.supabase.co/rest/v1/leads
# Hoje: retorna todos os leads
# Após esta story: retorna 401 Unauthorized
```

Débitos remediados: **TD-C3** (parcial — RLS ativo fecha o acesso anônimo), **TD-C4** (RLS com auth.uid() válido).

## Dev Notes

### SQL completo da migration

```sql
-- supabase/migrations/002_security_foundation.sql
-- DEPENDÊNCIA: 001_initial_schema.sql deve existir primeiro
-- DEPENDÊNCIA: Supabase Auth deve estar configurado (Story 1.1.1)
-- RISCO: Execute APENAS após confirmar que auth.uid() retorna UUID válido

-- 1. Habilitar RLS em todas as tabelas
ALTER TABLE leads           ENABLE ROW LEVEL SECURITY;
ALTER TABLE tasks           ENABLE ROW LEVEL SECURITY;
ALTER TABLE events          ENABLE ROW LEVEL SECURITY;
ALTER TABLE call_logs       ENABLE ROW LEVEL SECURITY;
ALTER TABLE okrs            ENABLE ROW LEVEL SECURITY;
ALTER TABLE content_posts   ENABLE ROW LEVEL SECURITY;
ALTER TABLE store_metrics   ENABLE ROW LEVEL SECURITY;

-- 2. Policy single-user (owner_all) em todas as tabelas
-- Permite qualquer operação para qualquer usuário autenticado
DO $$
DECLARE
  t text;
BEGIN
  FOREACH t IN ARRAY ARRAY[
    'leads','tasks','events','call_logs',
    'okrs','content_posts','store_metrics'
  ]
  LOOP
    EXECUTE format(
      'CREATE POLICY "owner_all" ON %I FOR ALL USING (auth.uid() IS NOT NULL)',
      t
    );
  END LOOP;
END $$;
```

[Source: docs/architecture/db-specialist-review.md#32-migration-002--security-foundation]

### Por que `auth.uid() IS NOT NULL` e não um UUID específico

Opção A (mais restritiva): `auth.uid() = 'uuid-do-carlos'::uuid` — só o Carlos acessa
Opção B (usada aqui): `auth.uid() IS NOT NULL` — qualquer autenticado acessa

Usamos Opção B porque:
1. Mais simples de implementar
2. Se Carlos criar outro usuário no futuro, não precisa alterar as policies
3. O sistema já é single-tenant por design (URL privada, sem cadastro público)

Se quiser Opção A no futuro, é uma migration simples: `DROP POLICY "owner_all"; CREATE POLICY "owner_only" ...`

### Risco de quebra total

**NUNCA executar esta migration sem Story 1.1.1 completa.**

Se RLS for habilitado sem auth ativo:
- `auth.uid()` retorna `NULL`
- Policy `auth.uid() IS NOT NULL` avalia como `FALSE`
- **TODOS** os dados ficam inacessíveis — o app quebra completamente

**Rollback de emergência:**
```sql
-- Se o app quebrar após a migration:
ALTER TABLE leads           DISABLE ROW LEVEL SECURITY;
ALTER TABLE tasks           DISABLE ROW LEVEL SECURITY;
ALTER TABLE events          DISABLE ROW LEVEL SECURITY;
ALTER TABLE call_logs       DISABLE ROW LEVEL SECURITY;
ALTER TABLE okrs            DISABLE ROW LEVEL SECURITY;
ALTER TABLE content_posts   DISABLE ROW LEVEL SECURITY;
ALTER TABLE store_metrics   DISABLE ROW LEVEL SECURITY;
```

### Proteção de PII pós-RLS

O campo `call_logs.transcript_snippet` contém transcrições de conversas (PII). Após esta migration, ele só será acessível por sessões autenticadas. Isso é o mínimo necessário.

No futuro (Sprint 2+), pode-se adicionar policy mais restritiva para este campo específico ou encrypt-at-rest.

[Source: docs/architecture/db-specialist-review.md#agr-1]
