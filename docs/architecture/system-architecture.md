# System Architecture — Nexus OS
> Generated by @architect (Brownfield Discovery Phase 1) — 2026-03-16

## Overview

**Nexus OS** é um dashboard executivo SPA (Single Page Application) para gestão multi-negócio de um CEO. Consolida OKRs, CRM de leads, gestão de tarefas, criação de conteúdo, rotina, calls de vendas e IA em uma interface unificada.

**Contexto de Negócio:**
- Unidades: 3D Digital, Grupo VcChic, VcChic, Mivave, Sezo, Moriel, Personal
- Produtos: Nexus, Mapa da Clareza, Formação 3D, Projeto Respirar
- Usuário primário: Carlos (CEO)

### Data Flow (Current)
```
User Action → Page Component → window.dispatchEvent('nexus-data-updated')
                                      ↓
                              App.tsx fetchData()
                                      ↓
                         7x Supabase queries (parallel)
                                      ↓
                              setState → re-render
```

## Database (Supabase)

**Connection:** `lib/supabase.ts` — client direto com anon key

**Tabelas identificadas:**
| Tabela | Descrição |
|--------|-----------|
| `leads` | CRM: prospects e clientes |
| `tasks` | Tarefas com regra 1-3-5 (Big Rock/Medium/Small) |
| `events` | Calendário e agenda |
| `call_logs` | Registro de calls de vendas/mentoria |
| `okrs` | Objectives & Key Results por unidade |
| `content_posts` | Posts para LinkedIn/Instagram/Newsletter |
| `store_metrics` | Métricas de e-commerce (ROAS, vendas, spend) |

---

## AI Layer

**Gemini Live (VoiceAssistant.tsx):**
- Modelo: `gemini-2.5-flash-native-audio-preview-12-2025`
- Conexão: WebSocket bidirecional (audio streaming)
- Input: 16kHz PCM mono
- Output: 24kHz PCM (voz sintetizada)
- Function Calling: `log_sales_call` → persiste CallLog

**Perfis de classificação de leads (sistema prompt):**
1. "Resgate" — Endividado/urgência → Projeto Respirar
2. "Transição" — CLT estável/liberdade → Formação 3D

## Critical Issues (Brownfield Discovery)

### CRITICAL — Segurança
| ID | Issue | Localização |
|----|-------|------------|
| CRIT-1 | Supabase URL e anon key hardcoded | `lib/supabase.ts:4-5` |
| CRIT-2 | GEMINI_API_KEY exposta no bundle client-side | `vite.config.ts:15-16` |

### HIGH — Arquitetura
| ID | Issue | Impacto |
|----|-------|---------|
| HIGH-1 | God Component: App.tsx com estado global total | Manutenibilidade, testabilidade |
| HIGH-2 | Sem service layer (queries direto no componente) | Acoplamento, sem reutilização |
| HIGH-3 | Sem React Router (navegação via useState) | Sem deep linking, sem histórico do browser |
| HIGH-4 | Event bus via window.dispatchEvent (não tipado) | Frágil, sem garantias |

### MEDIUM — Qualidade
| ID | Issue | Localização |
|----|-------|------------|
| MED-1 | `any` nos data mappers | `App.tsx:60,73,83,95,109` |
| MED-2 | StoreMetric interface inline | `App.tsx:21-27` → mover para `types.ts` |
| MED-3 | Mix dados mock + dados reais | `constants.ts` vs Supabase |
| MED-4 | Sem ESLint/Prettier | — |
| MED-5 | Zero testes | — |

## Recommended Target Architecture

```
src/
├── features/          # Feature-based modules
│   ├── dashboard/
│   ├── crm/           # Pipeline + Leads
│   ├── okrs/
│   ├── tasks/
│   ├── calls/
│   ├── routine/
│   ├── content/
│   └── ai-advisor/
├── services/          # Supabase service layer
│   ├── leads.service.ts
│   ├── tasks.service.ts
│   └── ...
├── store/             # Zustand global state
│   └── app.store.ts
├── components/        # Shared UI components
├── lib/               # Config (supabase, etc.)
└── types/             # Global types
```

**State Management:** Zustand (leve, TypeScript-first, sem boilerplate)
**Routing:** React Router v7 (compatible com React 19)
**Env:** Mover credentials para `.env.local`
